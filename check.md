# Reveal Overlapping Windows — 의논 체크리스트

목표: 현재 "Reveal Stacked Windows"(같은 크기의 같은 자리 창만 후보)를 확장/대체해서
**활성 창과 겹치는 모든 창**을 후보로 보여주는 기능을 만든다.

각 항목을 하나씩 결정하면서 체크해 나간다. 개발은 모든 항목 결정 + 사용자 명시 승인 이후에만 시작.

---

## [x] 1. "겹친다"의 정의

활성 창 A와 다른 창 B가 "겹친다"고 판정할 기준.

- [ ] A. 1픽셀이라도 교차 — `A.intersects(B)` (너무 느슨)
- [ ] B. 활성 창 면적 기준 — `A∩B / A면적 ≥ 임계값`
- [ ] C. 활성 창에 포함되는 창만 — `A.contains(B)` 또는 그 역 (너무 좁음)
- [ ] D-min. 양방향 모두 의미있게 — `min(A∩B/A, A∩B/B) ≥ 임계값`
- [x] **D-max. 어느 한쪽이라도 의미있게** — `max(A∩B/A, A∩B/B) ≥ 임계값`

**결정**: D-max
**임계값**: 0.25 (25%)
**근거**: 작은 떠다니는 창(예: 1Password mini, 색상 피커)의 절반이 큰 활성 창에
걸쳐 있는 케이스도 "겹친 창"으로 인식되길 원함. max를 쓰면 그 작은 창 입장에서
50% 겹친 것이 임계값 통과하므로 후보로 잡힘.
임계값 25%는 모서리만 살짝 닿은 우연한 겹침은 거르되, 작은 창이 자기 면적의
1/4 이상 활성 창과 걸치면 후보에 포함되도록 한 균형점.

## [x] 2. 활성 창 자신 포함 여부

현재 코드: `if w == activeWindow { continue }`로 제외.

- [x] **그대로 제외** — 후보 리스트는 "다른 창들"만
- [ ] 포함 — 활성 창도 후보에 넣고 cursor가 거기 시작

**결정**: 제외 유지
**근거**: 이 액션의 목적은 "겹친 창들 사이에서 전환"이므로 활성 창은 후보에
넣을 필요가 없다. 기존 Reveal Stacked Windows 동작과 일관성 유지.

---

## [x] 3. 같은 디스플레이 제약

현재 코드: 같은 `NSScreen`만 후보로 인정.

- [x] **명시적 제약 제거** — 좌표 겹침(D-max, 25%)만으로 판정. 디스플레이 경계에
      걸친 창도 자연스럽게 포함됨.
- [ ] 유지 — 같은 디스플레이 창만

**결정**: 제약 제거
**근거**: D-max + 25% 임계값이 이미 충분히 엄격하므로 디스플레이 단위의 추가
제약은 불필요하고 오히려 D-max 정의와 모순됨(좌표상 25% 겹쳤는데 디스플레이가
다르다는 이유로 제외하면 일관성 무너짐). 멀티 모니터에서 경계에 걸친 창도
자연스럽게 후보로 잡힘.

---

## [x] 4. 결과 정렬 순서

현재: front-to-back (`CGWindowList` 순서).

- [x] **front-to-back 유지** — 사용자에게 익숙한 z-order
- [ ] 겹침 비율 높은 순 — A와 가장 많이 겹친 창부터
- [ ] 다른 기준

**결정**: front-to-back 유지
**근거**: 기존 StackedWindowsPickerWindow UX와 일관성. picker 호출 의도는
보통 "최근 작업한 창들 사이 전환"이며 z-order가 그 의도에 자연스럽게 맞음.
매번 정렬이 바뀌면 muscle memory가 안 생기는 단점도 회피.

---

## [x] 5. 액션 교체 vs 추가

현재 액션: `revealStackedWindows` (같은 frame 창만).

- [x] **기존 액션 동작 자체를 교체** — `revealStackedWindows`가 이제 "겹친 창
      (D-max, 25%)" 의미가 됨. 같은 크기/같은 자리 케이스도 자연스럽게 포함됨.
- [ ] 별도 액션 추가 — `revealOverlappingWindows`(가칭) 새로 만들고 기존은 유지
- [ ] 기존 액션은 deprecate, 새 액션 표준화

**결정**: 기존 액션 동작을 교체
**근거**: 사용자 의도가 "기존 reveal window 기능에서 동일한 크기가 아닌 D-max
임계값 0.25 크기의 화면도 선택할 수 있도록 하는 기능"이라고 명시함. 즉
기존 액션의 확장이지 새로운 액션이 아님. 동일 크기 케이스는 임계값 25%를
자동으로 통과하므로 기존 사용자의 동작은 손실되지 않음(superset).
